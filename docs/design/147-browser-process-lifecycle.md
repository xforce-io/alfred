# #147 浏览器进程生命周期治理

最后更新：2026-07-12

## 问题

`twitter-watch` 依赖 `web` 技能的共享浏览器服务，但调用方可以通过任意 shell 命令检查端口、杀死 Node/Chromium 并重建服务。进程所有权不清晰会放大抓取失败，并可能干扰并发调用方。

## 设计

`skills/web/server.sh` 是唯一生命周期入口，提供 `start`、`status`、`restart`、`stop`。变更操作使用文件锁串行化，PID 文件只记录入口自身启动的进程；停止操作只作用于经过命令行校验且仍属于 web server 的 PID。

`start-server.ts` 只负责启动服务。CDP 端口被未知进程占用时返回明确错误，不再执行 `kill -9`。

`twitter-watch/fetch_tweets.py` 只做健康检查；服务不存在时调用一次 `server.sh start --headless`，失败即返回结构化文本，不自行枚举或终止进程。

## 测试计划

- 单元测试：生命周期命令解析、PID 所有权校验、未知端口占用失败。
- 集成测试：并发启动只有一个 owner；抓取启动失败不会杀进程或无限重试。
- 功能测试：服务已运行和服务未运行两条抓取入口均按预期工作。
- 端到端测试：启动服务、执行一次抓取、通过唯一入口重启、再次抓取，验证调用方没有终止非 owner 进程。

## 非目标

不引入新的 daemon 级浏览器平台，不改变 Playwright profile 和抓取逻辑。
